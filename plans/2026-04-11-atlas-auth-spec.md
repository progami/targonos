## Confirmed Issues
1. Local Atlas signed-out redirect and callback generation are configured to production origins, not the local Atlas/portal topology. Atlas middleware builds the login URL with `portalUrl('/login', request)` and the callback with `resolveAppOrigin(request)` ([apps/atlas/middleware.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/middleware.ts:85)), while Atlas local env pins `NEXTAUTH_URL`, `NEXT_PUBLIC_APP_URL`, `NEXT_PUBLIC_PORTAL_AUTH_URL`, and `PORTAL_AUTH_URL` to `https://os.targonglobal.com...` ([apps/atlas/.env.local](/Users/jarraramjad/dev/targonos-main/apps/atlas/.env.local:15)). `resolveAppOrigin()` does not fall back to request headers; the unit test explicitly expects it to throw when those env vars are absent ([apps/atlas/lib/request-origin.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/request-origin.ts:11), [apps/atlas/tests/unit/middleware-origin.test.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/tests/unit/middleware-origin.test.ts:26)). This matches the existing smoke finding that local auth drift is currently driven by production callback/domain values ([plans/2026-04-11-cross-app-ci-smoke-spec.md](/Users/jarraramjad/dev/targonos-main/plans/2026-04-11-cross-app-ci-smoke-spec.md:48)).
2. Atlas edge auth and Atlas server/API auth do not inspect the same session-cookie set. Middleware probes `targon`, `atlas`, and legacy Atlas `hrms` cookie names ([apps/atlas/middleware.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/middleware.ts:47)), but `getCurrentUser()` only probes `targon` and `atlas` ([apps/atlas/lib/current-user.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/current-user.ts:191)). That divergence matters because SSO still carries legacy Atlas entitlement normalization from `hrms` to `atlas` ([apps/sso/app/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/sso/app/page.tsx:30)).
3. Dev auth bypass is implemented with conflicting rules between the edge and the shared auth/session layer. Atlas middleware bypasses auth whenever either bypass flag is set in non-production, with no localhost guard ([apps/atlas/middleware.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/middleware.ts:35)). The shared auth package only enables bypass when those flags are set and the context is localhost-like ([packages/auth/src/index.ts](/Users/jarraramjad/dev/targonos-main/packages/auth/src/index.ts:866)). `getCurrentUser()` does not pass a request object into `decodePortalSession()`, so it cannot use request host data to satisfy that localhost check ([apps/atlas/lib/current-user.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/current-user.ts:199), [packages/auth/src/index.ts](/Users/jarraramjad/dev/targonos-main/packages/auth/src/index.ts:348)).
4. Atlas route entrypoints do not fail auth the same way once middleware and server-side session resolution disagree. `/hub` is server-gated and redirects straight to `/no-access` when `getCurrentEmployeeId()` is null ([apps/atlas/app/(atlas)/hub/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/hub/page.tsx:5)), but `/tasks`, `/tasks/add`, and `/tasks/[id]` are client routes that render first and then surface API failures from `TasksApi`, `ensureMe()`, and `EmployeesApi.listManageable()` ([apps/atlas/app/(atlas)/tasks/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/tasks/page.tsx:52), [apps/atlas/app/(atlas)/tasks/add/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/tasks/add/page.tsx:58), [apps/atlas/app/(atlas)/tasks/[id]/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/tasks/[id]/page.tsx:145)).
5. Atlas API auth responses are not standardized, and the client forwards those raw messages to users. Middleware returns `Authentication required` or `No access to Atlas` with a `reason` field for denied API requests ([apps/atlas/middleware.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/middleware.ts:69)), but downstream handlers return `Unauthorized`, `Unauthorized - not logged in`, or a 409 provisioning message depending on the route ([apps/atlas/app/api/me/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/me/route.ts:12), [apps/atlas/app/api/tasks/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/tasks/route.ts:31), [apps/atlas/app/api/tasks/[id]/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/tasks/[id]/route.ts:55), [apps/atlas/app/api/work-items/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/work-items/route.ts:11), [apps/atlas/app/api/dashboard/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/dashboard/route.ts:13), [apps/atlas/app/api/employees/manageable/route.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/employees/manageable/route.ts:15)). `request()` in `lib/api-client.ts` throws `body.error` directly, so these inconsistent strings become user-visible behavior ([apps/atlas/lib/api-client.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/api-client.ts:104)).
6. `/no-access` hardcodes the portal back-link to production instead of using the shared portal-origin helper, so the one Atlas route specifically intended for auth/entitlement recovery exits local/CI environments incorrectly ([apps/atlas/app/no-access/page.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/no-access/page.tsx:86), [apps/atlas/lib/portal.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/portal.ts:8)).

## Likely Root Causes
- Atlas auth topology is drifting across files: Atlas local env points auth/public URLs at production ([apps/atlas/.env.local](/Users/jarraramjad/dev/targonos-main/apps/atlas/.env.local:15)), Atlas package dev script binds `3006` ([apps/atlas/package.json](/Users/jarraramjad/dev/targonos-main/apps/atlas/package.json:6)), root dev app config maps Atlas to `3206` ([dev.local.apps.json](/Users/jarraramjad/dev/targonos-main/dev.local.apps.json:5)), and CI placeholder env uses `3106`/`3100` ([apps/atlas/.env.dev.ci](/Users/jarraramjad/dev/targonos-main/apps/atlas/.env.dev.ci:10)).
- Auth decisions are duplicated instead of shared: middleware calls `requireAppEntry()` with one cookie set and one bypass rule ([apps/atlas/middleware.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/middleware.ts:46)), while `getCurrentUser()` separately decodes cookies with a different set and without request context ([apps/atlas/lib/current-user.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/lib/current-user.ts:187)).
- Client pages are using API auth as a second gate instead of one consistent route-level decision. `/tasks*` depend on client fetches after render, `/hub` does a server redirect, and the shared layout suppresses `/api/me` failures by “default to non-admin” ([apps/atlas/app/(atlas)/layout.tsx](/Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/layout.tsx:289)).
- Existing browser auth coverage is not asserting the actual handoff contract. The signed-out smoke only checks for portal UI ([apps/atlas/tests/e2e/smoke.spec.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/tests/e2e/smoke.spec.ts:9)), authenticated Atlas tests skip when Google SSO cannot be automated ([apps/atlas/tests/e2e/smoke.spec.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/tests/e2e/smoke.spec.ts:15), [apps/atlas/tests/fixtures/auth.ts](/Users/jarraramjad/dev/targonos-main/apps/atlas/tests/fixtures/auth.ts:19)), and the cross-app smoke spec already calls out that this is not a hard-fail auth gate ([plans/2026-04-11-cross-app-ci-smoke-spec.md](/Users/jarraramjad/dev/targonos-main/plans/2026-04-11-cross-app-ci-smoke-spec.md:43)).

## Recommended Fixes
- Pick one canonical Atlas dev/CI origin and use it everywhere: Atlas `.env.local`, portal app-map resolution, test env, and callback generation. Atlas should not point its local auth/public env vars at production when the app is being exercised locally.
- Unify session-cookie probing across middleware and `getCurrentUser()`. Either keep legacy `hrms` support everywhere until it is proven dead, or remove it everywhere in one pass after confirming SSO/session emitters no longer use it.
- Remove Atlas-specific bypass branching from middleware and let the shared auth package own bypass rules end-to-end. The edge, `getCurrentUser()`, and `requireAppEntry()` should all make the same bypass decision.
- Standardize Atlas auth API responses: 401 for unauthenticated, 403 for entitled-session failures, and 409 only for confirmed employee-profile provisioning state. Align `/api/me`, `/api/tasks*`, `/api/work-items*`, `/api/dashboard`, `/api/employees/manageable`, and `/api/access-requests` to the same contract.
- Make `/hub` and `/tasks*` converge on one failure path for auth/session mismatch. The current split between server redirect and post-render API errors will keep producing different UX for the same underlying auth defect.
- Replace the hardcoded `/no-access` portal link with the same portal-origin helper the middleware already uses.

## Open Questions
- Is legacy `hrms` session/entitlement data still emitted anywhere that Atlas must support? Code preserves it, but there is No direct evidence yet from a live cookie capture in this review.
- Which Atlas local origin is canonical for shared-auth work: `http://localhost:3006/atlas`, `http://localhost:3206/atlas`, or CI’s `http://localhost:3106/atlas`? Current repo files disagree.
- Should an authenticated user with no employee record land on `/no-access`, get a 409 provisioning response, or be sent back through portal recovery? Current Atlas routes mix all three outcomes.
- Should dev auth bypass be supported outside localhost-like environments? There is No direct evidence yet that non-localhost bypass is a real requirement.
