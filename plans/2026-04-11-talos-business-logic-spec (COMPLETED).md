## Likely Root Causes
- Shipment planning was partially decommissioned: the consumer page was left in place, the live fetch was removed, and the remaining action path still points at a generic PO screen instead of a shipment-plan contract.
- The inventory page does not trust the ledger contract and re-derives semantic sign from the last movement type, which is incompatible with a net-balance payload.
- Transaction detail and transaction API drifted apart: the page assumes a normalized cargo-line structure that the API no longer supplies.
- The incomplete queue is keyed to “missing metadata at creation time,” while reconciliation is keyed to “required attachments currently present,” so the remediation list and the reconciliation state can diverge.
- GRN creation, warehouse receiving, and transition gate requirements were implemented as separate workflows without a single authoritative step-order.
- PI handling evolved in layers: legacy header fields remained, line-level PI numbers drive documents, and a newer dedicated PI table/API was added without becoming the sole source of truth.

## Recommended Fixes
- Restore one shipment-planning data contract. Either reinstate a reviewed inventory-comparison API or make shipment planning read the synced `sku.fbaStock` data directly, and carry selected SKUs into a concrete create-plan/create-PO flow instead of a blind redirect.
- Render inventory balances directly from the ledger’s net `current*` fields. Keep `lastTransactionType` only for badges or contextual labels, not for the numeric sign.
- Normalize transaction detail data at the API boundary. Return line items for receive/ship transactions in the shape the page expects, or suppress the cargo tab until that data exists.
- Make `/operations/inventory/incomplete` select by actual incompleteness, not only missing header fields. Attachment completeness and `isReconciled` should participate in the query predicate.
- Pick one GRN workflow. Either auto-create/update a GRN when PO inventory is received, or keep GRN as a separate explicit step and stop using `grnNumber` as a warehouse-stage primary identifier until that step completes. The transition gate and warehouse UI need to follow the same order of operations.
- Consolidate PI state behind one authority, most likely `purchaseOrderProformaInvoice` rows. The PO detail flow, document requirements, and stage transition payloads should all read and write the same PI model.

## Open Questions
- Should warehouse receiving auto-create a GRN record, or is GRN intentionally a separate operator action after receipt?
- Which PI model is intended to be authoritative: order header fields, per-line `piNumber`, or `purchaseOrderProformaInvoice` rows?
- Is shipment planning supposed to produce a purchase-order draft, a shipment draft, or only a filtered handoff into the existing PO workflow?
- The current app tree has no standalone shipment or invoice page. Is that intentional consolidation, or are those routes meant to exist but are currently missing?
