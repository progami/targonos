## Repro Routes
- `GET /policies` as HR or super-admin.
- `POST /api/policies` via `/policies/add` with `status=DRAFT`.
- `GET /performance/reviews/[id]` for a review currently in `NOT_STARTED`.
- `PATCH /api/performance-reviews/[id]` with a `status` field.
- `PATCH /api/disciplinary-actions/[id]` or save from `/performance/violations/[id]` after changing status in the form payload.
- `POST /api/leaves` for an employee whose `reportsToId` is `null`.
- `POST /api/uploads/finalize` after obtaining a presigned upload but without confirming the object was actually written.
- `POST /api/employees/[id]/offboard`, then open the generated task at `/tasks/[id]`.

## Confirmed Issues
- Opening the policy list performs a write mutation. [policies/page.tsx](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/policies/page.tsx:63>) automatically calls the admin consolidation endpoint on mount, and [company-wide/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/admin/policies/conduct/company-wide/route.ts:11>) executes [ensureConductPolicyCompanyWide()](</Users/jarraramjad/dev/targonos-main/apps/atlas/lib/domain/policies/conduct-company-wide.ts:28>), which updates the canonical policy to `region='ALL'`, archives other active conduct policies, and copies acknowledgements. A read route should not silently rewrite policy state.
- Creating a draft policy still triggers a “published” broadcast. The add form defaults to `status='DRAFT'` in [policies/add/page.tsx](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/policies/add/page.tsx:32>), but [policies/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/policies/route.ts:126>) always creates a `POLICY_CREATED` notification with the title `New Policy Published` and message `has been published` at [lines 140-151](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/policies/route.ts:140>). Draft creation is causing a publish side effect.
- Opening a `NOT_STARTED` performance review mutates it to `IN_PROGRESS`. [performance/reviews/[id]/page.tsx](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/performance/reviews/[id]/page.tsx:115>) auto-calls `PerformanceReviewsApi.start(id)` when `raw.status === 'NOT_STARTED'` at [lines 153-162](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/performance/reviews/[id]/page.tsx:153>), and [start/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/performance-reviews/[id]/start/route.ts:75>) persists `status='IN_PROGRESS'` plus `startedAt` and an audit log. Record viewing is changing workflow state.
- Generic performance-review PATCH bypasses the explicit approval workflow. [performance-reviews/[id]/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/performance-reviews/[id]/route.ts:95>) accepts arbitrary review updates and directly persists `status` at [line 186](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/performance-reviews/[id]/route.ts:186>), even though the app already has dedicated transition routes such as `start`, `submit`, `hr-review`, `super-admin-review`, and `acknowledge`. This lets callers skip the intended state machine.
- Generic disciplinary-action PATCH bypasses the explicit approval workflow. The violation edit form includes `status` in its editable schema and saves the full form payload back through the API in [performance/violations/[id]/page.tsx](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/performance/violations/[id]/page.tsx:57>) and [lines 225-241](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/performance/violations/[id]/page.tsx:225>), while [disciplinary-actions/[id]/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/disciplinary-actions/[id]/route.ts:82>) directly persists `status` at [line 140](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/disciplinary-actions/[id]/route.ts:140>). That bypasses the dedicated HR review, super-admin review, acknowledge, and appeal routes.
- Leave creation can strand requests in `PENDING_MANAGER` with no approver path. [leaves/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/leaves/route.ts:138>) always creates the request with `status='PENDING_MANAGER'` at [lines 272-282](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/leaves/route.ts:272>), but only creates a notification if `employee.reportsToId` exists at [lines 303-326](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/leaves/route.ts:303>). For employees without a manager, the request is initialized into the manager stage without assigning any approver.
- Upload finalization can create document metadata for a file that does not exist in storage. [uploads/finalize/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/uploads/finalize/route.ts:44>) validates the key pattern and permissions, then immediately writes `employeeFile` or `caseAttachment` rows at [lines 72-86](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/uploads/finalize/route.ts:72>) and [128-140](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/uploads/finalize/route.ts:128>) without any object existence check against the uploaded key. That breaks document lifecycle integrity.
- Offboarding tasks persist `actionUrl` values that Atlas task screens do not expose. [employees/[id]/offboard/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/employees/[id]/offboard/route.ts:10>) generates actionable offboarding links such as Google Admin, SSO admin, and employee profile URLs at [lines 28-46](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/employees/[id]/offboard/route.ts:28>), and [tasks/route.ts](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/tasks/route.ts:145>) persists `actionUrl` at [line 149](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/api/tasks/route.ts:149>). But the task detail form schema, reset state, and update payload in [tasks/[id]/page.tsx](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/tasks/[id]/page.tsx:18>) and [lines 149-157](</Users/jarraramjad/dev/targonos-main/apps/atlas/app/(atlas)/tasks/[id]/page.tsx:149>) never surface that field, so the workflow stores actionable links that users cannot execute from the task UI.

## Likely Root Causes
- Read surfaces are mixing display logic with hidden repair/migration logic.
- Generic CRUD endpoints were left open to status mutation after explicit workflow transition endpoints were added.
- Notifications and other side effects are keyed to record creation, not to validated state transitions.
- The document upload contract trusts client-declared success instead of verifying storage state before committing metadata.
- `actionUrl` was added to task persistence/template generation without a matching task-surface consumer.
- The spec/test-plan route list has drifted from the actual Atlas app tree in this checkout.

## Recommended Fixes
- Remove automatic conduct-policy consolidation from `/policies`; run it only through an explicit admin action or one-time migration.
- Gate policy notifications on actual publish/activate transitions, and use status-aware message copy.
- Remove auto-start from the review detail page; require an explicit start action before mutating review state.
- Reject `status` in generic `PATCH /api/performance-reviews/[id]` and `PATCH /api/disciplinary-actions/[id]`; force all state changes through the dedicated workflow endpoints.
- On leave creation, hard-fail if the employee has no manager, or route directly to the correct next approver stage. Do not initialize a manager-stage request without a manager.
- In upload finalize, perform a storage existence check against the uploaded key before writing any `employeeFile` or `caseAttachment` row.
- Either expose `actionUrl` prominently in task list/detail surfaces or stop generating/storing it until the task UI can execute it.
- Reconcile the test plan against the actual Atlas route tree so future workflow investigations start from live surfaces, not stale paths.

## Open Questions
- Are the requested `/documents` and `/templates` routes from a different Atlas branch or an older implementation? They are not present in the current `apps/atlas/app` tree.
- Should Atlas have one canonical offboarding workflow? The employee-profile modal performs full offboarding, while `/onboarding` is checklist-oriented.
- For employees without a manager, should leave requests hard-fail, jump directly to HR, or jump directly to super-admin?
- Should task `actionUrl` remain a generic task capability across Atlas, or is it only intended for onboarding/offboarding checklists?
